# 멋사 백엔드 스터디 · 모니터링 & 알림 편

> 지난 편에서 배포는 완전 자동이 됐습니다. push만 하면 서버에 알아서 배포되죠.
> 그런데 생각해보면 무섭습니다. **앱이 새벽 3시에 조용히 죽으면, 우리는 그걸 어떻게 알까요?**
> 사용자가 "사이트 안 열려요" 하고 알려줄 때까지 모른다? 그건 서비스가 아니죠.
>
> 오늘은 **서버와 앱 상태를 자동으로 감시**하고, **문제가 생기면 디스코드(또는 슬랙)로 알림이 오게** 만듭니다. 목표: **앱이 죽으면 30초 안에 내 폰이 띠링 울리기.** 

---

## Part 0. 오늘 할 일 지도

```
[감시자] ──매 30초마다──> [우리 앱 상태 확인]
   │                          │
   │                    정상? → 조용히 넘어감
   │                    죽음? → [디스코드/슬랙으로 알림 발사] ──> 📱 내 폰 띠링!
```

오늘의 흐름:

1. 우리 앱에 "나 살아있어요" 하고 대답하는 **헬스체크 엔드포인트** 만들기
2. 이 상태를 **자동으로 확인**하는 감시자 붙이기
3. 문제 생기면 **디스코드로 알림** 보내기

> 처음엔 "간단한 스크립트 + 알림"으로 원리를 이해하고, 마지막에 "제대로 된 모니터링 도구"도 소개할게요. 원리를 먼저 손으로 겪어보는 게 이해가 빨라요.

---

## Part 1. 헬스체크 엔드포인트 만들기 (앱이 대답하게)

감시를 하려면 앱한테 "너 살아있니?" 물어볼 창구가 필요해요. 이걸 **헬스체크(health check) 엔드포인트**라고 합니다. Spring에는 이게 **이미 기능으로 들어있어요.** (Actuator)

> 여기서 잠깐 — 이건 새 Spring 문법을 배우는 게 아니라, **이미 아는 Spring을 운영 관점에서 다시 보는** 첫 지점이에요. 앞으로 이런 순간이 종종 나옵니다.

### ① 의존성 추가

`build.gradle`에 Actuator 추가:

```gradle
dependencies {
    // ... 기존 것들
    implementation 'org.springframework.boot:spring-boot-starter-actuator'
}
```

### ② 헬스 엔드포인트 노출 설정

`application.yml` (또는 `.properties`)에:

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health    # health 엔드포인트만 외부에 노출
```

### ③ 확인

앱을 재배포하고 나면 이 주소가 살아있는지 응답합니다.

```bash
# 서버에서 확인 (앱 컨테이너가 살아있으면 UP이 뜸)
curl http://localhost:8080/actuator/health
```

정상이면 이런 응답이 옵니다.

```json
{"status":"UP"}
```

> 이 `{"status":"UP"}` 이 바로 "나 살아있어요"라는 앱의 대답이에요. 앱이 죽으면 이 주소는 응답을 아예 안 하거나 `DOWN`을 뱉습니다. 감시자는 이걸 보고 판단해요.

## Part 2. 디스코드 알림 창구 만들기 (Webhook)

알림을 어디로 받을지 정해야죠. 스터디에선 **디스코드 웹훅**이 제일 쉽고 무료예요. (슬랙도 방식은 거의 같아요 — 뒤에 언급)

### 디스코드 웹훅 만들기

1. 알림 받을 디스코드 서버에서 채널 하나 준비 (예: `#서버-알림`)
2. 채널 옆 **⚙️ 톱니바퀴(채널 편집)** → **연동(Integrations)** → **웹후크(Webhooks)**
3. **새 웹후크** → 이름 정하고 → **웹후크 URL 복사**

복사한 URL은 이렇게 생겼어요:

```
https://discord.com/api/webhooks/1234567890/abcdefg...
```

> 이 URL로 메시지를 POST하면 그 채널에 글이 올라옵니다. 테스트해볼까요?

```bash
# 이 URL로 메시지 하나 쏴보기 (URL은 본인 것으로)
curl -H "Content-Type: application/json" \
  -d '{"content":"테스트 알림입니다 🔔"}' \
  "https://discord.com/api/webhooks/..."
```

디스코드 채널에 "테스트 알림입니다 🔔"가 뜨면 성공! 이제 이 창구로 "앱 죽었다"는 알림을 보낼 거예요.

> 웹훅 URL은 **비밀**입니다. 이걸 아는 사람은 누구나 그 채널에 글을 쓸 수 있어요. 코드에 직접 넣거나 깃에 커밋하지 말고, 지난 편처럼 `.env`나 환경변수로 관리하세요.

## Part 3. 감시 스크립트 만들기 (원리 이해용)

이제 "매 30초마다 앱 상태를 확인하고, 죽었으면 디스코드로 알림"을 하는 감시자를 만듭니다. 먼저 **직접 스크립트로** 만들어서 원리를 이해할게요.

### ① 스크립트 작성

서버에서:

```bash
nano ~/healthcheck.sh
```

내용 (웹훅 URL은 본인 것으로):

```bash
#!/bin/bash

WEBHOOK="https://discord.com/api/webhooks/..."
URL="http://localhost:8080/actuator/health"

# 앱 상태 확인 (응답 코드가 200이면 정상)
STATUS=$(curl -s -o /dev/null -w "%{http_code}" "$URL")

if [ "$STATUS" != "200" ]; then
    # 정상이 아니면 디스코드로 알림
    curl -s -H "Content-Type: application/json" \
      -d '{"content":" 앱이 응답하지 않습니다! (상태코드: '"$STATUS"')"}' \
      "$WEBHOOK"
fi
```

저장: `Ctrl + O` → `Enter` → `Ctrl + X`

```bash
# 실행 권한 주기
chmod +x ~/healthcheck.sh
```

> 이 스크립트가 하는 일: 앱 헬스체크 주소에 요청 → 응답 코드가 200(정상)이 아니면 → 디스코드로 알림 발사. 딱 이게 전부예요. 모니터링의 본질이 이렇게 단순합니다.

### ② 자동 반복 설정 (cron)

이 스크립트를 **매분 자동 실행**되게 등록합니다. cron은 리눅스의 "정해진 시간마다 자동 실행" 스케줄러예요.

```bash
# cron 편집기 열기
crontab -e
```

(처음이면 편집기 고르라고 나옴 → nano 추천, 번호 선택)

맨 아래에 추가:

```
* * * * * ~/healthcheck.sh
```

> `* * * * *` 는 "매분 실행"이라는 뜻이에요. 저장하면 이제 서버가 알아서 1분마다 앱을 확인합니다.

### ③ 테스트 — 진짜 알림 오나 확인

일부러 앱을 꺼봅시다.

```bash
# 앱 컨테이너만 잠깐 중지
docker compose stop app
```

1분 안에 디스코드에 알림이 뜨면 **성공!** 확인했으면 다시 켜주세요.

```bash
docker compose start app
```

> 이 순간이 오늘의 하이라이트예요. **내가 아무것도 안 보고 있어도, 앱이 죽으면 폰이 울린다.** 이게 "서비스를 운영한다"는 감각의 시작입니다.

## Part 4. 한 단계 위 — 제대로 된 모니터링 도구 (소개)

Part 3의 스크립트는 원리를 이해하기엔 완벽하지만, 실제로는 이런 걸 대신해주는 **전용 도구**들이 있어요. 스터디원 상황에 맞게 골라 쓰면 됩니다.

### 옵션 A. Uptime Kuma (자체 호스팅, 추천)

우리 서버에 도커로 하나 더 띄우는 **무료 오픈소스 모니터링 대시보드**예요. 웹 화면에서 클릭 몇 번으로 감시 대상·알림을 설정할 수 있고, 예쁜 그래프와 가동률(uptime %)도 보여줍니다.

```yaml
# docker-compose.yml에 서비스 하나 추가하는 느낌 (개념)
  uptime-kuma:
    image: louislam/uptime-kuma:1
    restart: unless-stopped
    volumes:
      - uptime_data:/app/data
    # Caddy 뒤에 붙여서 status.내도메인.com 같은 걸로 접근
```

> 웹 UI에서 "이 URL을 30초마다 확인, 죽으면 디스코드로 알림" 을 클릭으로 설정. Part 3에서 손으로 한 걸 GUI로 편하게 하는 것 + 그래프까지. **스터디에서 실습으로 하나 띄워보기 딱 좋습니다.**

### 옵션 B. UptimeRobot (외부 서비스)

가입만 하면 되는 **외부 무료 모니터링 서비스**. 우리 서버 밖에서 감시하기 때문에 **서버 자체가 통째로 꺼져도** 알림이 온다는 장점이 있어요. (자체 호스팅은 서버가 죽으면 감시자도 같이 죽는 맹점이 있음)

> **실무 팁:** 이 두 개는 성격이 달라요. Uptime Kuma는 "우리 서버 안에서 세밀하게", UptimeRobot은 "밖에서 서버 생사 자체를". 둘을 **같이** 쓰면 서로의 맹점을 메웁니다. 스터디에선 Uptime Kuma로 실습하고, UptimeRobot을 "밖에서 보는 보험"으로 추가하는 구성을 추천.

## Part 5. 슬랙으로 받고 싶다면?

디스코드 대신 슬랙을 쓴다면 방식은 거의 같아요.

1. 슬랙 → **Incoming Webhooks** 앱 추가 → 채널 선택 → 웹훅 URL 복사
2. 보내는 형식만 살짝 다름 (`content` 대신 `text`):

```bash
curl -H "Content-Type: application/json" \
  -d '{"text":"앱이 응답하지 않습니다!"}' \
  "https://hooks.slack.com/services/..."
```

> Uptime Kuma를 쓴다면 알림 종류에서 디스코드/슬랙/텔레그램/이메일 등을 클릭으로 고르기만 하면 돼서 이 차이도 신경 쓸 필요 없어요.

## Part 6. 자주 나는 에러 & 해결법

###  `curl http://localhost:8080/actuator/health` 가 404

**원인:** Actuator 의존성 추가 후 재배포를 안 했거나, `application.yml`의 노출 설정을 안 함.
**해결:** Part 1의 ①②를 확인하고 `docker compose up -d --build`로 재배포.

###  디스코드 테스트 메시지가 안 옴

**원인:** 웹훅 URL 오타 또는 JSON 형식 문제.
**해결:** URL을 다시 복사해서 확인. `-d '{"content":"..."}'` 의 따옴표가 깨지지 않았는지 확인 (특히 복붙할 때 스마트따옴표 주의).

###  cron이 스크립트를 실행 안 함

**원인:** 실행 권한 누락, 또는 경로 문제.
**해결:**
- `chmod +x ~/healthcheck.sh` 했는지 확인.
- crontab에서 `~` 대신 **절대경로**(`/home/ubuntu/healthcheck.sh`)로 쓰는 게 더 안전. cron은 `~`를 못 알아들을 때가 있음.
- 로그 확인: `grep CRON /var/log/syslog`

###  앱을 껐는데 알림이 안 옴

**원인:** cron 주기가 아직 안 돌았거나(최대 1분 대기), 스크립트 내부 웹훅 URL이 틀림.
**해결:** 스크립트를 직접 한 번 실행(`~/healthcheck.sh`)해서 알림이 오는지 먼저 확인. 스크립트 단독은 되는데 cron만 안 되면 위 cron 문제로.

###  알림이 1분마다 계속 옴 (스팸)

**원인:** 앱이 계속 죽어있으면 매분 알림이 옴.
**해결:** 이건 사실 정상 동작이지만 시끄럽죠. 실무에선 "상태가 바뀔 때만 알림"(죽었을 때 1번, 살아났을 때 1번)을 구현하거나, Uptime Kuma 같은 도구가 이걸 알아서 처리해줍니다. 스터디에선 "그래서 전용 도구를 쓰는구나"를 느끼는 포인트로.

## 오늘의 3줄 요약

1. **모니터링의 본질 = "앱한테 살아있냐 묻고, 대답 없으면 알린다".** Spring Actuator의 `/actuator/health`가 앱의 "나 살아있어요" 대답 창구.
2. **알림 = 웹훅 한 방.** 디스코드/슬랙 웹훅 URL로 메시지를 POST하면 채널에 뜬다. cron으로 매분 자동 확인 → 죽으면 발사.
3. **실무는 전용 도구로.** 원리는 스크립트로 겪어보고, 실제 운영은 Uptime Kuma(안에서 세밀하게) + UptimeRobot(밖에서 서버 생사) 조합을 추천.

이제 여러분 앱은 **혼자서도 자기 상태를 감시하고, 문제가 생기면 먼저 알려줍니다.** 진짜 "운영되는 서비스"에 한 발 더 가까워졌어요.
